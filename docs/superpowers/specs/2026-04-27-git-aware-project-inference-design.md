# Git-Aware Project Inference

Status: Design approved 2026-04-27.

## Goal

Let Mostly CLI (and, later, other clients) figure out the active project, task, and actor from the git repository the user is currently in, so commands like `mostly task create --title 'x' --type chore` work without `--project AUTH` when run inside a linked repo.

Inference is graceful: if we can't infer everything, we infer what we can. Project is the most important; task is best-effort; actor falls back to `default_actor`.

## Non-goals

- MCP server integration. The resolution endpoint is server-side so MCP can adopt later, but no MCP changes ship in v1.
- Inferring from anything other than git (no shell history, no editor state).
- Multi-email per principal. A single nullable `email` column is enough for v1.
- Windows-specific path quirks. The project is Linux-first.
- Auto-creating projects from unknown repos.

## Mapping model

A project owns a set of repo links. A repo link is `(normalized_url, subpath)`. A single project can have many links (multiple remotes, forks, monorepo subdirs). The same `(normalized_url, subpath)` cannot link to two projects.

**URL normalization:** lowercase host, strip leading `git@`/`ssh://`/`https://`/`http://`, replace the `:` between host and path in SCP-style URLs with `/`, strip trailing `.git`, strip trailing `/`, drop fragment/query. Ports are preserved. Resulting form: `host[:port]/owner/repo` (e.g. `github.com/acme/auth`). Submodule URLs are ignored — only top-level remotes are read.

**Subpath:** the repo-relative directory the link applies to. Empty string = repo root. Normal git subpath inference uses `path.relative(repoRoot, cwd)`. The matching rule is **longest-subpath-prefix wins**; ties at the same length across different projects are an ambiguity error.

## Data model

| Column         | Type | Notes                                                      |
|----------------|------|------------------------------------------------------------|
| id             | text | ULID, PK                                                   |
| workspace_id   | text | FK → workspace                                             |

## Resolution logic

A new module `packages/core/src/git-context.ts` exposes:

```ts
gatherGitContext(cwd: string): GitContext | null
// Reads: rev-parse --show-toplevel, remote -v, rev-parse --abbrev-ref HEAD, config user.email.
// Returns null when not in a git repo.

resolveProjectFromGit(client, gitCtx): { project, link } | null
// POSTs urls + relPath to /v0/git-context/resolve. Server applies longest-subpath rule.

inferTaskFromBranch(branch: string, projectKey: string): string | null
// First match of /(<KEY>)-(\d+)/ where group 1 === projectKey (case-sensitive).

resolveActorFromEmail(client, email): Principal | null
// Looks up principals by email within the workspace.
```

`null` everywhere = "couldn't tell" → caller falls back to explicit flag, default, or error as appropriate.

`GitContext` shape:

```ts
type GitContext = {
  repoRoot: string;
  remotes: { name: string; normalized_url: string }[];  // origin first
  relPath: string;       // path.relative(repoRoot, cwd), '' at root
  branch: string | null; // null on detached HEAD
  authorEmail: string | null;
};
```

The git runner is a thin wrapper around `child_process.execFile('git', ...)` with a fake-able interface for unit tests. No third-party git library — keeps the dep surface flat.

## CLI surface

### New commands

- `mostly project link [--project <KEY>] [--subpath <PATH>] [--remote <NAME>|--all-remotes] [--from <PATH>]`
  - Default `--remote origin`. With `--all-remotes`, links every remote.
  - `--subpath` defaults to `""` (repo root). Users with a monorepo set it explicitly. Defaulting to the cwd's relative path was considered and rejected: it would silently scope a link to a subdir for any user who happens to run the command from inside one.
  - Without `--project`: prints the candidate normalized URLs and exits non-zero. (No interactive prompt — keeps it agent-friendly.)
  - Idempotent: re-linking the same `(url, subpath)` to the same project prints `already linked` and exits 0. Re-linking to a *different* project is a 409 error.

- `mostly project unlink [--project <KEY>] [--remote <NAME>] [--subpath <PATH>] [--all]`
  - Removes one link (default: `origin` at current subpath) or every link on the project with `--all`.

- `mostly project links [--project <KEY>]`
  - Lists links. Without `--project`: lists every link in the workspace (admin view), useful for debugging "where is this remote already linked?".

- `mostly whereami`
  - Read-only diagnostic. Prints what would be inferred for `cwd`: matched project + link, inferred task (if any), inferred actor (if any), and the signal each came from. Does not call any mutating endpoint.

### Implicit inference on existing commands

Affected commands: anything that takes `--project`, `--task`, or `--actor` (`task create`, `task list`, `task show`, `task claim`, `task start/close/...`).

Resolution per flag, in order:

1. Explicit flag (`--project AUTH`).
2. Inferred from git context.
3. Existing default (e.g. `default_actor` for actor).
4. Error if still required.

Inference is on by default. `--no-git-context` (one flag covering all three signals) disables it for the invocation. The flag is added to the same option set as the existing `--actor` etc. on every affected command.

A one-line stderr note when inference contributes anything: `(inferred: project=AUTH, task=AUTH-1, actor=eran)`. Suppressed by `--quiet` or `--json`.

## HTTP API

All endpoints under `/v0`, authenticated against the existing workspace auth.

- `POST /projects/:id/repo-links` — body `{ normalized_url, subpath }`. Returns the created link. 409 if `(workspace, url, subpath)` already linked anywhere.
- `DELETE /projects/:id/repo-links/:linkId` — removes one link.
- `GET /projects/:id/repo-links` — list links for a project.
- `GET /repo-links` — list all links in the workspace (used by `mostly project links` without a project flag).
- `POST /git-context/resolve` — body `{ urls: string[], rel_path?: string }`. Returns `{ project, link } | null`. Single resolution endpoint the CLI calls before mutations. Server applies the longest-subpath-wins rule and ambiguity check.

Authorization: link write endpoints (POST/DELETE) require write access to the workspace; link read endpoints (GET) and the resolve endpoint require any authenticated principal in the workspace.

## Error handling and edge cases

| Situation | Behavior |
|---|---|
| Not in a git repo | Inference returns `null`; commands behave as today. No warning unless `--verbose`. |
| Git repo with no remotes | Same — `null`, silent fallback. |
| Remotes present, none link to a project | `null`. Commands that require a project include the hint: `no project linked for <repo>; run "mostly project link --project <KEY>"`. |
| Multiple remotes resolve to different projects | Hard error pre-mutation: `ambiguous: origin → AUTH, upstream → BILLING. Pass --project explicitly or use --no-git-context.` |
| Multiple subpath links match `cwd` | Longest subpath wins. Tie at same length across projects → ambiguity error. |
| Branch yields a task ID for the wrong project | Branch hint is silently ignored — branches don't switch projects. |
| Branch matches `<KEY>-<digits>` but task doesn't exist | Inference returns the key anyway; the API call fails with the normal 404. No pre-validation round trip. |
| `git config user.email` matches multiple principals | Fall back to `default_actor`; one-line stderr note: `(actor not inferred: 2 principals match <email>)`. |
| `git config user.email` matches zero principals | Fall back silently to `default_actor`. |
| Detached HEAD | No branch → no task inference. Project + actor still work. |
| Worktrees | Treated like normal checkouts; `git rev-parse --show-toplevel` and `git remote -v` already do the right thing. |
| Server unavailable when resolving | Inference fails closed: `null`, single-line stderr note. Doesn't block commands that have explicit flags. |
| `--no-git-context` set | Skip everything: no git invocations, no resolve call. |
| Stale link (project archived) | Resolve treats it as no match; admin prunes via `mostly project links` + `unlink`. |
| Same `(url, subpath)` re-linked to same project | 200/idempotent: `already linked`. |
| Same `(url, subpath)` re-linked to *different* project | 409 from server. |
| Subpath on link doesn't currently exist on disk | Allowed; warning only. Lets users link a planned monorepo subdir. |
| Submodule URLs | Not read. Only top-level remotes are inspected. |

## Testing

### Unit tests (no git, no network)

- `packages/core/__tests__/git-context.test.ts`
  - URL normalization table: SCP-style ≡ HTTPS ≡ host/owner/repo, casing, trailing slashes, ports, `.git` suffix.
  - `inferTaskFromBranch`: positive (`AUTH-1-foo`, `feature/AUTH-1`, `eran/AUTH-1-foo`), negative (`main`, `auth-1`, key mismatch), null branch.
  - `resolveProjectFromGit` against a fake link store: longest-subpath-wins, multi-URL match, ambiguity, no-match.

- `packages/core/__tests__/git-runner.test.ts`
  - The wrapper around `git` is exercised via a fake; real `git` is covered in integration tests.

### Integration tests (real git, real server)

- `packages/cli/__tests__/git-context.integration.test.ts`
  - Tmp dirs, real `git init`, in-process server harness (mirroring the pattern in `packages/server/__tests__/`).
  - Cases: link → resolve → unlink round-trip; monorepo with two subpaths; ambiguous remotes; `--no-git-context`; server-down stub returning 503.

- `packages/server/__tests__/repo-links.test.ts`
  - Endpoint-level: auth, 409 duplicate, listing, longest-subpath rule, archived-project filtering.

### E2E

One scenario in `e2e/`: temp repo with origin pointing at a fixture URL, `mostly project link --project DEMO`, then `mostly task create --title 'x' --type chore` from inside the repo asserts the task lands in `DEMO`.

### Helpers

- `withTempGitRepo(opts)` in `@mostly/core/test-utils` returns `{ dir, run(cmd) }`.
- Extend `packages/core/src/test-utils/factories.ts` with `makeRepoLink({ project, normalized_url, subpath })`.

### Out of scope for v1 tests

MCP integration (no MCP wiring in v1), Windows path edge cases, exhaustive URL fuzzing.

## Rollout

- All changes ship together. The unique index and new endpoints are additive — existing CLI invocations with explicit flags keep working unchanged.
- Existing principals don't get an email until they re-run `mostly login`. Until then they don't participate in actor inference; this is intended.
- No data migration needed beyond the schema additions.
